         ↓
files 컬렉션에 레코드 INSERT (register_file 함수)

rag_file_ids 컬렉션 기록
         ↓
루프 종료 후 agents.tool_resources.file_search.file_ids 일괄 업데이트 (1회) ★
         ↓
다음 질문부터 에이전트가 File Search tool로 entity_id 필터링해서 검색

【★ Knowledge Layer 개념 - 주희 시스템 핵심 공백 ★】
===========================================

■ 배경: 안드레아 카파시 "LLM Wiki" 개념

전 테슬라 AI 책임자 카파시가 제시한 개인화 지식 관리 구조.
AI가 개인 데이터를 스스로 학습·가공하여 체계적 지식 저장소(Wiki)로
구축하는 시스템이며, 단순 RAG를 넘어선 능동적 지식 관리를 지향.

카파시 3계층 구조:
  레이어 1 (Raw Data): 원본 자료 보존 - 지식의 원천
  레이어 2 (Knowledge Layer): AI가 가공·분류·링크한 위키 문서층 ★ 핵심 ★
  레이어 3 (Operation Layer): AI 작동 방식 및 규칙 문서화

■ 주희 시스템 현재 상태

  레이어 1 (Raw Data): ✅ 100% 완성
    - MongoDB conversations 컬렉션
    - 109개 이상 대화 원본 저장
    - rag_file_ids 컬렉션으로 인덱싱됨
    - Phase 1에서 완성됨

  레이어 2 (Knowledge Layer): ❌ 20% (큰 공백)
    - 대화 원본은 있지만 가공된 위키 문서 없음
    - 문서 간 교차 링크 없음
    - RAG가 의미 유사도로만 검색 → 관계 구조 부재
    - 주희 시스템의 가장 큰 미완성 영역

  레이어 3 (Operation Layer): ⚠️ 40% (부분 완성)
    - 마스터 컨텍스트 파일 1개 존재
    - 에이전트 instructions 기본 설정됨
    - 상황별 의사결정 규칙 부재
    - 트리거 규칙 (언제 검색, 언제 기억 조회, 언제 실행) 체계화 안 됨

■ RAG vs Knowledge Layer 차이 (핵심)

  RAG (현재 방식):
    "도서관에서 그때그때 책 찾기"
    - 사용자 질문 시 의미 유사도로 관련 대화 검색
    - 검색 결과를 Claude가 실시간 종합
    - 문서 간 관계 구조 없음

  Knowledge Layer (카파시 제안):
    "도서관에 목차·인덱스·교차 링크 미리 구축"
    - AI가 사전에 정보 가공·분류
    - 개념 간 명시적 링크 생성
    - 주제별 정제된 위키 문서로 변환
    - 질문 시 이미 정리된 지식 즉시 활용

■ Knowledge Layer 구축 시 실제 변화 예시

  현재 RAG 작동:
    주희 질문: "코스피 인버스 전략 어떻게 했었지?"
    → RAG가 "코스피", "인버스" 키워드로 대화 검색
    → 유사도 높은 대화 5~10개 반환
    → Claude가 그때그때 종합
    → 결과: 매번 다를 수 있음, 맥락 일부 누락 가능

  Knowledge Layer 구축 후:
    사전 가공된 위키 문서:
      "코스피 인버스 전략 (2026-04 기준)"
        - 관련 대화 8개 링크 (시간순 정렬)
        - 주요 결정 사항: TSLS 진입 조건 3가지
        - 마스터 컨텍스트 연결: 휴전 랠리 시나리오
        - 실행 중인 포지션: ...
        - 관련 테마 링크: "해양 데이터센터", "반도체 실적"
    
    주희 질문 시:
    → 이미 정리된 위키 문서 즉시 참조
    → 일관된 답변 + 맥락 완결성
    → 관련 주제 자동 제시 ("해양 데이터센터 전략도 같이 보시겠어요?")

■ 문서 간 링크/교차 참조의 중요성

  카파시 핵심 제안:
    AI가 "이 대화는 저 대화와 연관된다"를 자동 링크 생성
    → 개념 간 관계망 구축
    → 주제별 맥락 완결성 확보

  주희 시스템 적용 시나리오:
    "코스피 인버스" 대화 ↔ "해양 데이터센터" 대화 ↔ "TSLS 진입" 대화
    → 공통 주제: 2026 매크로 전략
    → 자동 링크로 묶임
    → 주희가 한 주제 질문 시 관련 주제 자동 제시

■ 주희 시스템의 독자적 강점 (카파시 제안에 없는 것)

  강점 1: 멀티 툴 통합 구조
    - 카파시: ChatGPT + MCP 단일 인터페이스
    - 주희: LibreChat + File Search + web_search + YouTube/Reddit/네이버 등 다층

  강점 2: 액션 레이어 (Phase 3 예정)
    - 카파시: 지식 저장 + 호출 중심
    - 주희: OpenClaw + 자동매매 + Telegram 봇 → 실행 가능한 시스템

  강점 3: 자동화 파이프라인
    - 카파시: 수동 파일 업로드 + AI 가공
    - 주희: 1시간마다 자동 RAG 업데이트 (Phase 1에서 완성)

■ 주희 시스템 완성도 평가 (카파시 기준)

  Raw Data Layer:       ✅ 100%
  Knowledge Layer:      ⚠️ 20% ← 핵심 공백
  Operation Layer:      ⚠️ 40%
  Auto-update:          ✅ 100%
  Search/Retrieval:     ✅ 100%
  Multi-tool:           ✅ 우수 (카파시 제안 초과)
  Action Layer:         🚧 Phase 3 예정 (카파시 제안에 없음)

  종합: 카파시 수준의 60% + Action Layer로 일부 초과

■ Knowledge Layer 구축 도입 시점 판단

  옵션 A: Phase 2 완료 후 독립 Phase 2.5로 신설
    - 109개 대화를 주제별 위키 문서로 자동 변환
    - 문서 간 링크 자동 생성
    - 마스터 컨텍스트를 여러 계층으로 분할

  옵션 B: Phase 3에 통합
    - OpenClaw + 자동 브리핑 설계 시
    - Knowledge Layer 자동 생성 로직 포함
    - 새 대화 쌓일 때마다 위키 문서 업데이트

  옵션 C (권장): Phase 2 완료 후 재평가
    - 109개가 더 쌓이고 나서 위키화 효율적
    - Phase 3 OpenClaw가 이 역할 자연스럽게 담당 가능
    - 조기 구축 시 Phase 2 집중 분산 위험

■ Knowledge Layer 구축 시 기술적 접근 (개요)

  단계 1: 주제 분류
    - 기존 109개 대화를 Claude가 읽고 주제별 태그 부여
    - 예: #투자-전략, #음악-믹싱, #자동화-설계, #매크로-분석

  단계 2: 위키 문서 생성
    - 같은 주제 태그 대화들을 묶어 하나의 위키 문서로 합성
    - Claude가 핵심 요약 + 결정 사항 + 연관 정보 추출

  단계 3: 교차 링크 구축
    - 위키 문서 간 관련성 분석
    - 명시적 링크 메타데이터 생성
    - MongoDB 별도 컬렉션 또는 VectorDB에 저장

  단계 4: 자동 업데이트
    - 새 대화 추가 시 기존 위키 문서와 연결 판단
    - 필요 시 기존 문서 업데이트
    - OpenClaw가 이 역할 담당 가능

  단계 5: 에이전트 통합
    - File Search가 단순 대화 원본 → 정제된 위키 문서 우선 검색
    - 답변 품질 및 일관성 향상

■ 절대 잊지 말아야 할 점

  주희 시스템이 카파시 제안보다 "뒤처진" 게 아니다.
  
  실제 평가:
    - Raw Data + Auto-update + Multi-tool + Action Layer 관점에서는 카파시 초과
    - Knowledge Layer만 주희 시스템의 유일한 큰 공백
    - 이 공백이 메워지면 카파시 제안을 완전히 초월하는 시스템이 됨

  현재 109개 대화가 원본 상태로만 저장되어 있는 것이 아쉬운 지점.
  주제별 가공/링크만 추가되면 답변 품질이 질적 도약.

- 사용자 명시적 지시 없는 로드맵/전략 임의 추가 금지
- **LibreChat 신규 tool 추가 시 체크리스트:** tool 등록은 4곳 모두 필요 — ①manifest.json 엔트리 ②index.js require+export ③handleTools.js toolConstructors 매핑 ④packages/api/src/tools/registry/definitions.ts의 toolDefinitions record. 네 번째가 가장 쉽게 누락되는 숨은 요구사항. 누락 시 loadToolDefinitions가 silent drop → Claude API 페이로드 제외 → 에러 없이 tool 미실행.
