## Phase 1: RAG 자동저장 시스템 — 완전 완료 ✅

### 목표
LibreChat 대화를 자동으로 RAG에 저장하고, 에이전트가 File Search tool로 과거 대화를 검색할 수 있게 한다.

### 현재 상태
- 대화 **109개** RAG 임베딩 완료
- MongoDB `rag_file_ids` 컬렉션: file_id 매핑 저장
- MongoDB `files` 컬렉션: LibreChat File Search tool이 인식하는 레코드 동기화 완료
- MongoDB `agents` 컬렉션 `tool_resources.file_search.file_ids`: 전체 file_id 등록 완료
- 에이전트 File Search 검색 정상 동작 확인 ("4월 14일" 검색 → "여러 대화에서 알려주신 정보" 정확 응답)

### rag-autosave 스크립트
- 위치: Railway `rag-autosave` 서비스, GitHub `ldd0369/rag-autosave`
- 현재 버전: **v12**
- 실행 주기: 1시간마다 (`start.sh` while loop + sleep 3600)
- 환경변수: `MONGO_URI`, `RAG_API_URL`, `JWT_SECRET`, `FORCE_RESET`

### rag_autosave.py 동작 흐름
```
1. MongoDB rag_file_ids ↔ files 컬렉션 동기화 (시작 시 1회)
2. last_run 이후 새 메시지가 있는 대화 목록 조회
3. 대화별 전체 메시지 → 텍스트 변환
4. RAG API /embed (file_id, user, entity_id 포함)
5. rag_file_ids 컬렉션 upsert
6. files 컬렉션 upsert (LibreChat File Search 인식용)
7. 루프 종료 후 1회: agents 컬렉션 file_ids 업데이트
8. /query 테스트
9. rag_autosave_state last_run 갱신
```

## Phase 1 삽질 기록 — 진짜 근본 원인 3가지

### (a) AGENT_ID 오타
- 잘못된 값: `agent_cTUj86IrAnZ44EdmDB1iB`
- 정확한 값: `agent_cTUjB6IrAnZ44EdmDB1i8` ← **7번째 자리 B, 마지막 자리 8**
- 코드에서 AGENT_ID 상수가 정의만 되고 미사용이었던 것도 문제 (에이전트 이름으로 조회하다가 오타 노출 늦음)

### (b) LibreChat `files` 컬렉션 레코드 필수
- LibreChat File Search tool (`fileSearch.js`) 흐름:
  ```
  agents.tool_resources.file_search.file_ids 읽기
  → getFiles({ file_id: { $in: file_ids } })  ← MongoDB files 컬렉션 조회
  → 결과가 빈 배열이면 "No files to search" 반환
  ```
- `tool_resources`에 file_id를 등록하는 것만으로는 부족
- `files` 컬렉션에 아래 필드로 레코드 삽입 필수:

  | 필드 | 값 |
  |------|-----|
  | `user` | `ObjectId(USER_ID)` |
  | `file_id` | UUID |
  | `filename` | `conv_xxxx.txt` |
  | `filepath` | `"vectordb"` |
  | `bytes` | content byte 수 (기존분은 0으로 채움) |
  | `object` | `"file"` |
  | `type` | `"text/plain"` |
  | `embedded` | `True` |
  | `source` | `"vectordb"` |

### (c) RAG embed 시 `entity_id` 파라미터 필수
- LibreChat이 `/query` 호출 시 항상 `entity_id: agent.id` 포함 (`handleTools.js:315`)
- RAG API는 embed 시 받은 `entity_id`를 벡터 메타데이터에 저장하고, query 시 이것으로 필터링
- `entity_id` 없이 embed된 파일 → `entity_id`로 query하면 **0건 반환** (에러 없이 조용히 실패)
- rag_autosave.py 자체 `/query` 테스트는 `entity_id` 없이 호출 → 정상처럼 보여서 늦게 발견
- 수정: `/embed` 호출 시 `data`에 `entity_id=AGENT_ID` 추가

## 남은 작업

### P1 — 가까운 시일 내 처리

| 항목 | 내용 |
|------|------|
| `datetime.utcnow()` 제거 | `rag_file_ids` upsert, `rag_autosave_state` 갱신에 naive datetime 사용 중. Python 3.12+ DeprecationWarning. 전부 `datetime.now(timezone.utc)`로 통일 |

### P2 — 여유 있을 때 처리

| 항목 | 내용 |
|------|------|
| `sleep` 최적화 | 현재 대화당 `sleep(1)` → 109개 기준 ~109초 소요. 0.3초로 줄이거나 배치 조건부 적용 |
| 구 벡터 삭제 | 대화에 새 메시지 추가 시 전체 재임베딩하는데, 이전 file_id 벡터가 VectorDB에 누적됨. 재임베딩 전 `/documents` DELETE 호출 필요 |
| `initialize.js` 불필요 RAG 훅 제거 | LibreChat 내부에 대화 저장 시 자동 RAG 훅이 있을 경우, File Search tool이 이미 담당하므로 중복. 확인 후 제거 |

## Phase 2 후보 — 우선순위 제안

전체 자동화 로드맵 10개 기준, 다음 타깃 후보:

### 1순위: 카카오톡 대화 자동 수집 → RAG 저장
- **WHY**: 일상 대화, 약속, 업무 지시가 카카오에 집중되어 있음. RAG에 들어가면 에이전트 컨텍스트 폭발적으로 확장
- **HOW**: 카카오 백업 파일 파싱 → rag_autosave 파이프라인과 동일하게 embed
- **난이도**: 중 (백업 파일 포맷 파싱 + 주기적 수집 자동화)

### 2순위: 에이전트 기억 능동 업데이트 (Memory Write)
- **WHY**: 현재는 읽기(File Search)만 됨. 에이전트가 중요 정보를 스스로 메모할 수 있으면 능동형 비서로 진화
- **HOW**: 에이전트 system prompt에 "중요 정보 감지 → /memory endpoint 호출" 지시 추가. 별도 메모리 컬렉션 설계
- **난이도**: 중상

### 3순위: 일정/할일 자동 추출 → 캘린더 연동
- **WHY**: 대화에서 "이번 주 금요일 미팅" 같은 표현을 에이전트가 자동으로 캘린더에 등록
- **HOW**: RAG 저장된 대화에서 날짜/일정 패턴 추출 → Google Calendar API
- **난이도**: 중

### 4순위: 대화 요약 자동화 (Daily Digest)
- **WHY**: 109개 대화 전체가 벡터로 저장되어 있지만, 날짜별 요약이 있으면 검색 품질 향상
- **HOW**: 매일 자정 rag_autosave에서 당일 대화 요약 → 별도 summary 문서로 임베딩
- **난이도**: 하 (rag_autosave 파이프라인 재활용)

### 5순위: 외부 데이터 수집 자동화 (뉴스/메일)
- **WHY**: 에이전트가 외부 정보도 검색할 수 있게 되면 범용 비서로 확장
- **HOW**: Gmail API + 뉴스 RSS → 주기적 수집 → RAG embed
- **난이도**: 중
